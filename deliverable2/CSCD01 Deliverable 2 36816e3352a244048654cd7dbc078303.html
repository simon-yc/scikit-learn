It is designed to conveniently work together with Python libraries such as NumPy and SciPy.

Overall Architecture

Scikit-Learn follows a 2 layered architecture that includes the Base layer and the ML Models layer. The Base layer consists of generic interfaces for the Estimators, Predictors, and Transformers categories of Scikit-Learn, each of which have a concrete implementation provided in the ML Models Layer.

An estimator interface is used for creating and fitting models, a predictor interface for producing predictions, and a transformer interface for converting data. All objects within Scikit-Learn use this consistent common basic API.

Consider the following example of how the Base layer of Scikit-Learn interacts with the Model layer:

from sklearn import tree
X = [[0, 0], [1, 1]]
Y = [0, 1]
clf = tree.DecisionTreeClassifier()
clf = clf.fit(X, Y)

The above code snippet illustrates the relationship between the Base layer and the ML Model layer. Classes in the Model layer inherit from BaseEstimator in the Base Layer, which provides the basic implementation for all Estimators, including defining their parameters and providing setter and getter methods for the parameters.

Depending on whether the functionality is necessary or not, Estimator classes may also inherit from any of the Mixin classes provided by the Base layer. The DecisionTreeClassifier or DecisionTreeRegressor classes can ultimately be used by the end-user to carry out the proper classification or regression, respectively.

Base Layer

All ML models start with the base layer, which serves as the foundation for their behavior. The base layer is made up of classes that resemble interfaces and have a generic functions that invokes a particular function (e.g. fit, predict, etc.). The exact algorithm for machine learning that the model uses is then determined by implementing this function in a child class.

The base file includes two main types of classes classes, namely BaseEstimator and various mixins that child classes can use. In the Scikit-Learn project, nearly all classes either directly implement BaseEstimator or inherit it from a parent class since estimators are responsible for data fitting and training, which serve as the foundation for all machine learning algorithms. If needed by the class, the mixin classes offer extra functionality. For instance, classification algorithms utilize the ClassifierMixin to place data into categories, whereas regression algorithms use the RegressionMixin to predict continuous data. This approach encourages the design of modular classes and reduces code repetition because not all algorithms require the use of every mixin class.

ML Models Layer

The second layer in Scikit-Learn's architecture is the ML Model Layer, which comprises of distinct implementations of the various machine learning algorithms in Scikit-Learn. All of Scikit-Learn's features are handled by the various machine learning methods and features that are specific to each folder in the ML Models.

Based on its specific role inside Scikit-Learn, each class in the main folders is in charge of implementing the various Mixins and Interfaces in the Base layer. Problems and Possible Improvements in Scikit-Learn

  • Integration of deep learning: Despite the growing popularity of deep learning, Scikit-learn lacks built-in support for it. Users could take advantage of the advantages of both shallow and deep learning algorithms in a single, unified framework by integrating deep learning frameworks like TensorFlow, PyTorch, and Keras.
  • More thorough explanations: Although Scikit-Learn offers a bunch of documentation, some sections of the library might use more in-depth or user-friendly explanations. More examples, tutorials, and use cases could make it simpler for users to comprehend and make use of scikit-features.
  • The single responsibility principle is broken in some Scikit-Learn classes. For instance, both the fit() and predict() methods are present in the Predictor classes. Scikit-Learn uses these methods to represent two different functionality, so including both of them in the same class is against the rule. To prevent Predictors from extending Estimators, a potential solution would be to create distinct classes for each of the two functionalities.

Design Patterns

Design Pattern - Template Method

UML diagram:

Description:

Template method is defining an algorithm’s skeleton in an operation, and deferring some steps to subclasses. Template Method lets subclasses redefine certain steps of an algorithm without changing the algorithm's structure.

In the above case, the BaseRandomProjection class acts as the superclass and uses the Template Method design pattern to define the method fit() as a skeleton operation, where fit() relies on the abstract _make_random_matrix method. Inside the concrete subclasses GaussianRandomProjection and SparseRandomProjection, the method _make_random_matrix is implemented while still using the logic defined in the fit() method from the superclass. Therefore, as the subclasses each implement the abstract method without changing the overall structure of the method fit(), the Template Method Design Pattern is clearly being used.

Design Pattern - Decorator Method

UML Diagram:

Description:

Decorator method is used to add additional responsibilities to one object dynamically. It helps extend functionalities for subclasses.

As shown in the above graph: Pipeline class is the Decorator, Estimator class is the Component, AlphaParamPipline and RevParaamPiple classes are ConcreteDecorators used in test cases. Pipeline class takes a list of transform and estimator objects then applies them in sequence to the input data. All estimators have to implement the fit() method in order to be used by Pipeline class. Pipeline class is a decorator that adds additional abilities to Estimators dynamically. It allows clients to create complex machine learning models by providing a list of transforms and estimators.

For example: call Pipeline([('scalerA', estimatorA()), ('scalerB', estimatorB())])results apply estimatorA on input data getting data b, then apply estimatorB on data B getting output data.
